# 코스트코 제품 라벨 자동 OCR → 구글 시트 시스템 (Azure Read API 버전)

> 처음 이 저장소를 맡으셨다면 `HANDOFF.md`를 먼저 읽어주세요 — 전체 그림과
> 지금 뭘 해야 하는지가 정리되어 있습니다. 이 문서(README)는 시스템이 어떻게
> 동작하는지에 대한 상세 레퍼런스입니다.

Google Drive 폴더에 코스트코 매대 가격표 사진을 대량(예: 월 1,500장) 업로드하면
Azure AI Vision의 Read API로 무료 OCR을 돌려서, 상품코드/제품명(한국어)/가격만 뽑아
Google Sheets에 자동으로 정리해주는 시스템입니다. **대시보드의 "업데이트" 버튼을
누르면 그 시점에 실행**됩니다 (GitHub Actions의 `workflow_dispatch` API를 대시보드
백엔드가 호출하는 방식 - 연동 방법은 `DASHBOARD_INTEGRATION.md` 참고). 언제 사진이
올라올지 알 수 없는 상황에서 굳이 주기적으로 폴링할 필요가 없어서, 필요한 시점에만
실행되도록 했습니다. 서버 운영 비용 없이 완전 무료로 동작합니다.

## 왜 Azure Read API인가?

- 무료 티어(F0): **월 5,000건까지 완전 무료**, 월 1,500장 사용에는 비용이 전혀 들지 않습니다.
- 단, 무료 티어는 **분당 20건 속도 제한**이 있습니다. 1,500장 처리 시 약 75~90분 소요됩니다.
  (사람이 지켜볼 필요 없는 배치 작업이므로 문제되지 않습니다.)
- 유료로 전환하면 속도 제한이 크게 완화되지만, 지금은 무료 유지가 목표이므로 F0로 충분합니다.

## 왜 GitHub Actions인가?

- 서버를 따로 띄우지 않아도 **REST API 호출 한 번으로 실행을 트리거**할 수 있습니다
  (Private 저장소도 월 2,000분 무료 — 새 파일이 없으면 몇 초 만에 끝나는 실행이라 크게
  소모되지 않습니다. 사진을 한꺼번에 많이 올린 달에는 OCR 처리에 수십~90분 정도 쓰입니다).
- 처리 이력은 로컬 파일이 아니라 **구글 시트 자체**(파일ID 열)를 기준으로 판단하므로,
  매번 새로 뜨는 Actions 실행 환경에서도 중복 처리 없이 이어서 작동합니다.
- 워크플로에 `concurrency` 설정이 있어서, 이전 실행이 아직 진행 중일 때(대량 업로드
  직후 OCR이 오래 걸리는 경우) 버튼이 또 눌려도 두 실행이 겹치지 않고 순서대로 돕니다.

## 1. Azure 리소스 생성

1. https://portal.azure.com 접속 (없으면 무료 계정 생성, 카드 등록 필요하나 F0는 과금 없음)
2. "리소스 만들기" → "Azure AI Vision" 검색 → 만들기
3. 가격 책정 계층(Pricing tier)에서 **F0 (무료)** 선택 (구독당 F0 리소스는 1개만 생성 가능)
4. 생성 완료 후 리소스의 "키 및 엔드포인트" 메뉴에서 다음 두 값을 복사:
   - `AZURE_VISION_ENDPOINT` (예: `https://your-resource.cognitiveservices.azure.com`)
   - `AZURE_VISION_KEY`

## 2. Google 쪽 설정

1. [Google Cloud Console](https://console.cloud.google.com)에서 프로젝트 생성 후 **Drive API**, **Sheets API** 활성화
2. "API 및 서비스 → 사용자 인증 정보 → 서비스 계정 만들기"로 서비스 계정 생성 → JSON 키 다운로드
3. 결과를 저장할 Google Sheet를 하나 만들고, 다운로드한 JSON 안의 `client_email` 값을
   해당 시트에 **편집자**로 공유.
4. **Drive 폴더를 코스트코용/트레이더스용 두 개로 따로 만듭니다** (예: "코스트코 사진",
   "트레이더스 사진"). 사진 형식을 텍스트로 추측하지 않고 어느 폴더에 넣었는지로
   바로 판별하기 위해서입니다 - 코스트코 사진은 코스트코 폴더에, 트레이더스 사진은
   트레이더스 폴더에 넣어야 합니다. 두 폴더 모두 서비스 계정 이메일을 **편집자**로
   공유해야 합니다(처리 완료 사진을 하위 폴더로 이동하려면 뷰어 권한으로는 부족합니다).
5. 시트 URL에서 `SPREADSHEET_ID`, 두 Drive 폴더 URL에서 각각
   `DRIVE_FOLDER_ID_COSTCO` / `DRIVE_FOLDER_ID_TRADERS` 확인
   (`https://docs.google.com/spreadsheets/d/여기부분/edit`, `https://drive.google.com/drive/folders/여기부분`)

## 3. 로컬에서 테스트 실행

```bash
pip install -r requirements.txt
cp .env.example .env
# .env 파일에 DRIVE_FOLDER_ID_COSTCO, DRIVE_FOLDER_ID_TRADERS, SPREADSHEET_ID,
# AZURE_VISION_ENDPOINT, AZURE_VISION_KEY 채우기
# 다운로드한 서비스 계정 JSON을 service_account.json 이름으로 이 폴더에 저장

python main.py
```

## 4. GitHub Actions 실행 설정

저장소 **Settings → Secrets and variables → Actions → New repository secret**에서 아래 값을 등록하세요.

| Secret 이름 | 값 |
|---|---|
| `DRIVE_FOLDER_ID_COSTCO` | 코스트코 사진 전용 Drive 폴더 ID |
| `DRIVE_FOLDER_ID_TRADERS` | 트레이더스 사진 전용 Drive 폴더 ID |
| `SPREADSHEET_ID` | 구글 시트 ID |
| `SHEET_NAME` | 코스트코 시트 탭 이름 (예: `시트1`) |
| `TRADERS_SHEET_NAME` | 트레이더스 시트 탭 이름 (미설정 시 기본값 `트레이더스`) |
| `CATEGORY_SHEET_NAME_COSTCO` | 코스트코 제품군정리 시트 탭 이름 (미설정 시 기본값 `제품군정리(코스트코)`) |
| `CATEGORY_SHEET_NAME_TRADERS` | 트레이더스 제품군정리 시트 탭 이름 (미설정 시 기본값 `제품군정리(트레이더스)`) |
| `AZURE_VISION_ENDPOINT` | Azure 엔드포인트 URL |
| `AZURE_VISION_KEY` | Azure 키 |
| `GOOGLE_SERVICE_ACCOUNT_JSON` | 서비스 계정 JSON 파일의 **내용 전체**를 그대로 복사해서 붙여넣기 |
| `MC_CATALOG_SPREADSHEET_ID` | (선택) 자사 상품 카탈로그 구글 시트 ID — 7절 참고 |
| `MC_CATALOG_SHEET_NAME` | (선택) 카탈로그 시트 탭 이름 (미설정 시 기본값 `시트5`) |

### 자사 카탈로그 시트 준비 (제품군 자동 분류 정확도를 높이고 싶다면)

`MC_CATALOG_SPREADSHEET_ID`는 **이 저장소와는 별개의, 비공개 구글 시트**를
가리켜야 합니다. 이 저장소(`costco-label-ocr`)는 **Public**이라 카탈로그
데이터(거래처/공급사 이름 등 사외비 정보 포함)를 여기 코드나 파일로 직접
넣으면 안 됩니다.

1. 새 구글 시트를 만들고(기존 결과 시트와는 다른, 별도의 시트), `MCH1` /
   `MC` / `상품명` 3개 컬럼으로 자사 상품 카탈로그를 채웁니다 (`MCH1`은 대분류라
   실제 분류에는 안 쓰이지만 있어도 무방합니다. `MC`가 제품군, `상품명`이
   그 제품군에 속한 실제 SKU명입니다).
2. 이 시트를 서비스 계정 이메일(`GOOGLE_SERVICE_ACCOUNT_JSON`의 `client_email`)에
   **뷰어**로 공유합니다 (읽기만 하므로 편집자 권한은 필요 없습니다).
3. 시트 URL에서 `MC_CATALOG_SPREADSHEET_ID`를 확인해 시크릿으로 등록합니다.
   탭 이름이 `시트5`가 아니면 `MC_CATALOG_SHEET_NAME`도 같이 등록하세요.

설정하지 않아도 파이프라인은 정상 동작합니다 — 이 경우 제품군 분류는
`CATEGORY_KEYWORDS`(코드에 있는 수동 목록)만으로 이뤄집니다. 자세한 매칭
방식은 7절 참고.

등록 후에는 `.github/workflows/auto-ocr.yml`의 `workflow_dispatch` 트리거로 실행됩니다.
자동 스케줄(크론)은 없고, 아래 두 방법 중 하나로 실행을 트리거합니다:

- **대시보드 "업데이트" 버튼**: 대시보드 백엔드가 GitHub REST API를 호출 (연동 방법은
  `DASHBOARD_INTEGRATION.md` 참고)
- **수동 확인용**: 저장소의 **Actions 탭 → 코스트코 라벨 OCR 실행 (대시보드 트리거 / 수동 실행)
  → Run workflow**

## 5. 동작 방식

- 코스트코/트레이더스 두 Drive 폴더를 각각 전부 조회(페이지네이션 처리, 1,500장도
  누락 없이 조회)하고, 각 폴더에 대응하는 구글 시트의 `파일ID` 열에 이미 기록된
  파일은 건너뜁니다.
- 신규 이미지만 병렬로 다운로드하고, Azure 속도 제한(분당 18건, 여유 있게 설정)에 맞춰
  순차적으로 OCR을 호출합니다.
- 일시적 오류(네트워크 문제, 429 속도 제한 등)는 자동 재시도됩니다. 재시도 후에도 실패한
  파일은 시트에 기록되지 않으므로 다음 실행 때 자동으로 다시 시도됩니다.
- **사진 한 장당 항상 한 행만 추출합니다.** 프레임 안에 배경 진열대의 다른
  가격표가 흐릿하게 같이 찍혀도, 그 안에서 처음 인식되는 상품(주로 가장 크게/
  가장 먼저 나오는 목표 가격표)만 뽑고 나머지는 무시합니다. 상품을 여러 개
  등록하고 싶으면 사진을 상품별로 각각 따로 찍어서 올려주세요.
- **코스트코/트레이더스는 사진을 넣은 Drive 폴더로 구분합니다.** 예전에는 한
  폴더에 섞어 올리고 텍스트 내용(상품코드가 맨 앞 줄에 오는지 등)으로 자동
  판별했지만, 사진마다 레이아웃이 조금씩 달라서 오분류가 잦았습니다. 지금은
  `DRIVE_FOLDER_ID_COSTCO` 폴더에 넣은 사진은 무조건 코스트코 파서로, 
  `DRIVE_FOLDER_ID_TRADERS` 폴더에 넣은 사진은 무조건 트레이더스 파서로
  처리되어 오분류 자체가 구조적으로 발생할 수 없습니다. 코스트코 사진은
  `SHEET_NAME` 탭에, 트레이더스 사진은 `TRADERS_SHEET_NAME` 탭에 기록됩니다.
  두 탭 모두 없으면 자동으로 만들어집니다. 중복 처리 방지용 파일ID 체크는
  각 폴더/시트 안에서 따로 봅니다(폴더가 이미 분리되어 있으므로 두 탭을 합쳐
  볼 필요가 없습니다).
- **시트 기록이 끝난 사진은 삭제하지 않고 같은 Drive 폴더 안 `처리완료` 하위
  폴더로 이동합니다.** 원본은 그대로 보존되어 나중에 재검증이 필요하면 다시
  볼 수 있고, 신규 업로드 폴더(코스트코/트레이더스 각각)는 처리된 사진이
  빠지면서 계속 가벼운 상태로 유지되어 폴더 조회 속도가 데이터 누적과 무관하게
  일정합니다. 이 이동을 하려면 서비스 계정이 두 Drive 폴더 모두에 최소
  **편집자** 권한으로 공유되어 있어야 합니다(뷰어 권한만으로는 이동이 실패하며,
  실패해도 시트 기록은 이미 끝난 상태라 데이터 유실은 아니고 그 사진만 원래
  폴더에 남습니다).
- **소량 테스트나 특정 폴더만 처리하고 싶을 때**는 `workflow_dispatch` 호출
  시 `costco_folder_id` / `traders_folder_id` 입력값으로 각각 다른 Drive 폴더
  ID를 넘기면, 그 실행에서만 해당 시크릿 대신 그 폴더를 사용합니다(시크릿 값
  자체는 바뀌지 않음). 둘 다 비워두면 평소처럼 기본 폴더를 사용하고, 한쪽만
  채우면 그 리테일러만 다른 폴더로 테스트할 수 있습니다.
- **원본 시트(아래 6절)와 별개로, 제품군 단위로 정리된 시트도 같이
  갱신됩니다** (자세한 내용은 7절 참고).

## 6. 시트 컬럼 구조

**핵심 컬럼** (식품/비식품 가릴 것 없이 모든 SKU에 공통으로 채워짐):
`파일ID`, `파일명`, `처리일시`, `원문텍스트`, `상품코드`, `제품명(한국어)`, `가격`

**비핵심 컬럼** (사진에 그 정보가 있으면 채우고, 없으면 빈 칸):
`제품명(영어)`, `중량`, `단가`

핵심 컬럼을 앞에, 비핵심 컬럼을 뒤에 두는 순서입니다. 식품/비식품을 나누는
기준으로 컬럼을 다르게 쓰지 않고, 사진마다 있는 정보만 그때그때 채웁니다.

`파일ID`는 중복 처리 방지를 위한 식별자이며, `원문텍스트`에는 항목 파싱과 무관하게
OCR로 읽은 전체 텍스트가 그대로 남아있어 파싱이 잘못된 경우 원문 대조가 가능합니다.

`가격`은 정가/할인가가 같이 찍혀있어도 **항상 정가**를 선택합니다 ("할인행사" 문구나
"-4,500원" 같은 할인폭 표시가 나오면 그 이후에 나오는 가격은 절대 선택하지 않음).
`단가`는 "217원/10g"처럼 기준 단위까지 같이 기록해서, 몇 g당 가격인지 알 수 있게
합니다. 단가로 쓴 줄은 가격 후보에서 제외하므로 같은 값이 두 컬럼에 중복되지
않습니다.

`중량`은 "50G X 12", "1.2KG", "1.2kgx1개"처럼 숫자+단위(g/ml/kg/l/m) 패턴이면
공백 없이 곱셈 표시나 한글이 바로 붙어있어도 인식합니다.

## 7. 제품군정리 시트

코스트코/트레이더스 원본 시트(위 6절)와는 별개로, 같은 스프레드시트 안에
`CATEGORY_SHEET_NAME_COSTCO`(기본값 `제품군정리(코스트코)`)와
`CATEGORY_SHEET_NAME_TRADERS`(기본값 `제품군정리(트레이더스)`) 탭이 각각
하나씩 더 생깁니다 — 원본 시트가 코스트코/트레이더스로 나뉘어 있는 것과
마찬가지로, 상품 성격이 서로 달라서 제품군 정리도 리테일러별로 따로 둡니다.
원본 시트는 사진 한 장당 한 행씩 계속 쌓이는 로그 형태이고, 이 시트는 그
데이터를 **제품군(예: 올리브유) 단위로 묶어서 상품끼리 나란히 비교**할 수
있게 정리한 파생 뷰입니다. 원본 시트를 건드리지 않으므로, 이 시트의 분류가
잘못돼도 원본과 대조해서 언제든 다시 정리할 수 있습니다.

**구조** (카드형 - 상품 하나가 열 하나):

```
[제품군명]                         <- 제목 행
사진        (비어있음, 수기 채움용)
상품명      DR.G 닥터지 선크림      이니스프리 선크림
소싱형태    (비어있음)
매출(연)    (비어있음)
규격/단량   150ml                  50ml
판매가      26,990원               15,000원
단위단가    (비어있음)
매총율      (비어있음)
산도        (비어있음)
병입원산지  (비어있음)
셀링        - 자외선 차단 ...       - 산뜻한 사용감 ...

                                   <- 2행 공백
[다음 제품군명]
...
```

- OCR 상품카드에서 뽑히는 정보(상품명/규격·단량/판매가/단위단가/셀링)만
  채워지고, 나머지 행(사진/소싱형태/매출(연)/매총율/산도/병입원산지)은
  항상 빈 칸입니다 — 수기로 채우거나 다른 소스에서 채워 넣는 몫입니다.
  상품코드/파일ID/원문텍스트는 이 시트에는 없습니다 — 이 시트는 사람이
  보기 좋은 정리본이고, 그 추적용 정보는 원본(RAW) 시트에 사진 한 장당 한
  행씩 이미 남아있습니다.
- **규격/단량은 독립된 줄뿐 아니라 제품명 끝에 붙어 나온 경우도 자동으로
  분리합니다** ("오리온 오그래놀라 시나몬츄러스 440g* 3개" → 제품명 "오리온
  오그래놀라 시나몬츄러스" + 규격/단량 "440g* 3개").
- **셀링(셀링포인트)은 OCR 원문에서 `-`/`•`로 시작하는 문구를 뽑아 정리합니다**
  (`extract_selling_points()`). 화면 폭 때문에 한 문구가 여러 줄로 잘려
  나오면("- 신선하고 품질이 좋은 ... 압착하여" 다음 줄에 "만든 오일") 다음
  `-`가 나오기 전까지는 같은 문구로 이어붙입니다. 단, 가격/코드/영문제품명/
  "신세계포인트"·"적립"·"행사기간" 같은 정형 프로모션 문구가 중간에 끼면
  거기서 문구를 끊습니다 — 아직 완벽하지 않은 초기 버전이라 실사진으로
  계속 검증하며 다듬어야 합니다.
- **제품군 판별은 2단계입니다** (`detect_category()`):
  1. **자사 카탈로그 매칭** (`MC_CATALOG_SPREADSHEET_ID` 설정 시, 1순위):
     비공개 카탈로그 시트의 `상품명`들과 OCR 상품명 사이에서 겹치는 부분
     문자열을 찾습니다. 표기가 다 달라서(예: 카탈로그의 "OO브랜드 아보카도
     오일 1L"과 OCR의 "피에트로코리첼리 아보카도오일 1L(퓨어)") 완전일치가
     아니라 부분 문자열 겹침으로 찾되, 안전하게 매칭하기 위해 두 가지
     조건을 둡니다:
     (a) **가장 길게** 겹치는 부분 문자열부터 확인하고 (짧고 흔한 단어일수록
     오판 위험이 크므로), (b) 그 부분 문자열을 포함하는 카탈로그 항목들의
     제품군(MC)이 **한쪽으로 확실히 쏠려 있을 때만**(과반 이상, 최소 2건
     근거) 매칭을 확정합니다. 예를 들어 "아보카도"라는 흔한 단어만 겹치고
     카탈로그 안에서 "아보카도오일"/"아보카도"(과일) 등 여러 제품군에
     흩어져 있으면 억지로 어느 한쪽으로 정하지 않고 다음 단계로 넘어갑니다
     — 틀리게 분류되는 것보다 미분류로 남는 편이 낫다는 원칙입니다.
  2. **`CATEGORY_KEYWORDS` 수동 목록** (1단계에서 매칭 실패 시, 또는 카탈로그
     자체가 설정 안 돼있을 때): 상품명(한국어)에 이 키가 포함되어 있는지로
     판단하고, 매칭되면 매핑된 제품군 이름을 씁니다. 대부분은 키와 제품군
     이름이 같지만("올리브유"->"올리브유"), 제품명에 제품군을 나타내는
     글자가 아예 없는 경우를 위해 브랜드/제품종류 -> 실제 제품군 매핑도 같이
     둘 수 있습니다("골드베렌"->"구미젤리") — 다만 이런 브랜드들은 대부분
     카탈로그에 이미 있어서 카탈로그가 설정돼있으면 거의 쓰일 일이
     없습니다.

  두 단계 다 실패하면 `미분류` 블록에 모이므로 데이터가 유실되진 않습니다.
  카탈로그 데이터 자체의 `MC` 값에 거래처/공급사명 같은 게 섞여 있으면
  가끔 예쁘지 않은 제품군 이름이 나올 수 있는데, 이건 매칭 로직의 버그가
  아니라 카탈로그 데이터 품질 문제라 카탈로그를 정리하거나 수동으로
  고쳐줘야 합니다.
- 같은 상품코드가 다시 나와도(재촬영 등) 항상 새 열을 추가합니다 — 기존
  열을 덮어쓰지 않습니다.
- 병렬로 OCR 처리된 상품들을 시트에 배치하는 작업(제품군 블록에서 다음
  빈 열 번호를 정하는 것)은 여러 스레드가 동시에 하면 충돌하므로, 모든
  OCR이 끝난 뒤 한 번에 단일 스레드로 처리합니다.
- 이 시트는 아직 실사진으로 폭넓게 검증되지 않은 초기 버전입니다. 특히
  제품군 키워드 목록이 지금은 이 세션에서 실제로 봤던 몇 가지(선크림,
  올리브유, 구미젤리 등)만 들어있어서, 대부분의 사진이 처음엔 `미분류`로
  모일 수 있습니다.

## 8. 정확도 향상 팁

- Azure가 반환하는 줄 순서는 항상 사진의 실제 위→아래 순서와 일치하지는 않습니다
  (특히 상품코드/바코드 구역과 제품명/가격 구역이 서로 떨어져 있는 가격표 사진에서
  두드러짐). `ocr_image_azure()`가 각 줄의 bounding box 좌표로 자동 재정렬하므로
  대부분은 문제없지만, 그래도 이상하게 파싱되는 사진이 있으면 해당 행의 `원문텍스트`를
  확인해서 실제 줄 순서와 비교해보세요.
- 브랜드명이 "RICOLA", "MIKAKUTO"처럼 대문자로 단독 줄에 찍히는 경우가 많아서,
  `parse_price_fields()`의 영어 제품명 판별 조건은 "대문자 2글자 이상 연속 +
  2단어 이상"을 요구합니다. 브랜드명 한 단어로는 걸리지 않게 하기 위함입니다.
- 사진 촬영 시 빛 반사를 피하는 것이 어떤 OCR 엔진보다 정확도에 큰 영향을 줍니다:
  정면보다 살짝 비스듬히, 확산광(자연광)에서, 플래시 직사 없이 촬영하는 것을 권장합니다.
- 배경 진열대의 다른 가격표가 프레임에 같이 걸려도 시트에는 항상 한 행만
  추가되므로 걱정하지 않으셔도 됩니다(위 5절 참고). 다만 목표 가격표를 최대한
  크고 선명하게 찍을수록 그 한 행의 인식 정확도는 높아집니다.
- **사진에 다른 물체(옆 상품 박스, 뒤의 다른 가격표 등)가 같이 찍혀서 그
  텍스트까지 섞여 들어오는 문제를 줄이기 위해, 실제 사진 픽셀에서 배경색을
  뽑아 세로 줄 간격과 함께 레이아웃을 분석**합니다(`_cluster_lines_by_layout()`).
  같은 상품카드 안의 줄들은 촘촘하고 배경색도 일정하다고 보고, 간격이 크게
  벌어지거나(다른 물체로 전환) 간격이 살짝 벌어지면서 배경색도 확 다르면
  그 지점에서 나눠서, 가장 크게 찍힌(=실제 촬영 대상일 가능성이 높은) 덩어리만
  남깁니다. 이 분석은 실제 이미지 픽셀을 열어서 하는 거라 텍스트 기반
  수정들과 달리 **실사진으로만 진짜 정확도를 검증할 수 있습니다** - 두
  물체가 화면에서 거의 붙어있어서 간격 차이가 거의 없는 경우(예: 상품 박스
  바로 위에 가격표가 붙어있는 구조)는 이 방식으로도 완전히는 못 걸러낼 수
  있습니다. 이미지를 열지 못하거나 분석 중 오류가 나면 조용히 건너뛰고
  기존 방식(필터링 없음)으로 계속 진행합니다.
- **`parse_traders_fields()`는 실사진 검증이 아직 부족한 초기 버전입니다.**
  트레이더스 가격표 사진 1장(레이아웃 설명)만으로 만든 파서라, 코스트코
  파서가 겪었던 것과 같은 종류의 실사진 기반 버그(줄 순서, 브랜드명/단가
  오인식 등)가 나올 가능성이 높습니다. 트레이더스 사진을 실제로 여러 장
  돌려보고, `원문텍스트` 대비 잘못 뽑힌 필드가 있으면 코스트코 때와 같은
  방식으로 하나씩 고쳐나가야 합니다.

## 9. 헤더가 예상과 다를 때

시트 1행의 헤더가 위 컬럼 순서와 다르면, 자동으로 안전하게 처리 가능한 경우
(컬럼이 끝이나 중간에 새로 추가된 경우)만 알아서 맞추고, 그 외(순서 변경/삭제)는
경고만 출력하고 **기존 데이터는 지우지 않습니다**. 이 경우 시트 1행을 위 컬럼
순서에 맞게 직접 수정해주세요.
